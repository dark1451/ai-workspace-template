# 태스크 명확화 인터뷰

## 트리거

- 요구사항 해석이 둘 이상 가능할 때
- 기능 범위, 우선순위, UX 방향, 데이터 정책 등에서 **유저 결정**이 필요할 때
- 현재 정보만으로는 태스크를 구체화하거나 구현을 진행하면 오답 가능성이 높다고 판단될 때

## 원칙

- **인터뷰 전에는 진행하지 않는다**: 모호성이나 유저 결정 필요 사항이 있으면, 스펙·태스크 구체화나 구현을 계속하지 않는다.
- **질문은 짧고 구체적으로**: 무엇이 모호한지, 어떤 선택지가 있는지, 각 선택이 어떤 영향을 주는지 간단히 설명한다.
- **대기 상태를 명시한다**: 질문 후에는 반드시 **"현재 사용자 확인 대기 중"**이라고 알리고, 답변 전까지 다음 단계로 넘어가지 않는다.
- **결과는 문서에 반영한다**: 답변을 받으면 `docs/project-concept.md`와 관련 스펙/태스크 문서에 반영한다.
- 직전 질문이 하나의 승인 또는 소수의 명확한 선택지였다면, 사용자의 짧은 답변("응", "좋아요", "1안", "후순위로", "그 방향으로")도 현재 대기 중인 질문에 대한 유효한 응답으로 해석한다.

## 절차

1. **모호성 정리**
   - 무엇이 불명확한지 1~3개 정도의 질문으로 압축한다.
   - 가능하면 선택지를 함께 제시한다.
2. **영향도 설명**
   - 각 선택이 어떤 차이를 만드는지 한두 문장으로 설명한다.
3. **사용자 질문**
   - 예:
     - "로그인은 이메일만 지원할까요, 소셜 로그인도 포함할까요?"
     - "관리자 권한은 MVP에 포함할까요, 후속 단계로 미룰까요?"
4. **대기 안내**
   - 반드시 **"현재 사용자 확인 대기 중"**이라고 알린다.
   - 답변을 받기 전까지 태스크 생성, 스펙 확정, 구현을 진행하지 않는다.
5. **답변 반영**
   - 사용자의 결정을 `docs/project-concept.md`에 반영한다.
   - 관련 스펙, 태스크 문서도 함께 갱신한다.
   - 관련 태스크가 `waiting_user`였다면 현재 단계에 맞는 상태(`pending`, `in_progress`, `in_test`)로 되돌린다.
6. **재개**
   - 문서 반영 후에만 태스크 구체화 또는 구현을 재개한다.

## 완료 조건

- 모호했던 항목이 사용자 답변으로 해소되었다.
- 답변이 컨셉 문서와 관련 스펙/태스크 문서에 반영되었다.
- 이후 작업을 진행해도 된다는 기준이 확보되었다.
