# 동료 리뷰 (Peer Review)

## 트리거

- "리뷰해줘", "동료 리뷰 진행해줘", "코드 리뷰"
- 다른 에이전트가 작업 완료 후 리뷰를 요청한 경우

## 원칙

- **같은 역할의 다른 에이전트**가 리뷰한다 (기획자→기획자, 개발자→개발자, 테스트→테스트).
- 리뷰를 통과해야만 다음 단계로 상태를 전이할 수 있다.

## 단일 에이전트 환경 대응

Cursor에서 한 번에 하나의 에이전트만 동작하는 경우, "다른 에이전트"가 존재하지 않아 리뷰가 교착될 수 있다. 이때 다음 중 하나를 적용한다:

1. **새 세션 리뷰**: 작업한 세션을 종료하고, 새 채팅 세션에서 "리뷰해줘"를 호출한다. 새 세션은 기존 작업의 맥락 없이 산출물만 보고 리뷰하므로 일정 수준의 객관성이 확보된다.
2. **셀프 체크리스트**: 다른 세션도 불가능하면, 아래 체크리스트를 작성 에이전트가 직접 수행하고 `reviewNote`에 "셀프 리뷰"로 명시한다. 사용자가 결과를 확인하고 승인하면 상태를 전이한다.
3. **사용자 승인 리뷰**: 에이전트가 리뷰 결과를 사용자에게 요약 보고하고, 사용자가 "통과" 또는 "수정"을 지시한다.

셀프 리뷰 시 `reviewedBy`에 "self-review"로 기록하고, `reviewNote`에 체크리스트 결과를 포함한다. 향후 여러 에이전트가 동시에 동작하게 되면 반드시 다른 에이전트 리뷰로 전환한다.

## 리뷰 체크리스트

### 기획 산출물 (스펙·아이디어)
- [ ] 목표가 명확한가?
- [ ] 수용 조건이 구현자가 이해 가능한 수준인가?
- [ ] 태스크 분해가 적절한가?

### 개발·디자인 산출물 (코드·디자인)
- [ ] 스펙·디자인을 준수하는가?
- [ ] 코드 품질이 적절한가? (가독성, 구조)
- [ ] 단위/통합 테스트가 충분한가?
- [ ] handoff 메모(codePaths, testCommand 등)가 채워져 있는가?

### 테스트 산출물 (검증 결과)
- [ ] 검증 절차가 적절한가? (테스트 범위, E2E 시나리오)
- [ ] 결과 해석이 정확한가?
- [ ] testResult가 태스크 메모에 기록되었는가?

## 리뷰 결과 기록

태스크의 frontmatter 또는 `tasks/items/<id>.review.md`에 기록:

```yaml
reviewedBy: "(리뷰한 에이전트/세션)"
reviewResult: "pass"  # 또는 "request-changes"
reviewNote: "(통과 사유 또는 수정 요청 내용)"
```

## 수정 요청 시

- `reviewResult: request-changes`로 기록하고, `reviewNote`에 구체적인 수정 사항을 적는다.
- 작성 에이전트가 수정 반영 후 재리뷰를 요청한다.
- 재리뷰 시 이전 리뷰 코멘트가 반영되었는지 확인한다.
