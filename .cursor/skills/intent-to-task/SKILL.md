# 사용자 의도 → 태스크 변환

## 트리거

- "~ 하고 싶어", "이 기능 추가해줘", "기획해줘", "스펙 작성해줘"

## 절차

1. **프로젝트 컨셉 확인**: `docs/project-concept.md`가 있고, 프로젝트 개요·목표·대상·핵심 기능 등이 채워져 있는지 확인한다.
   - **없거나 비어 있으면** 스펙·태스크 생성을 진행하지 않고, "어떤 프로젝트를 진행하시겠습니까?"라고 묻고 **project-concept-interview** 스킬에 따라 컨셉 인터뷰를 진행한 뒤 문서를 작성·갱신한다. 컨셉 문서가 채워진 후에만 아래 단계로 진행한다.
2. **작업 진행 모드 확인**: `docs/project-concept.md`에 `proactive` 또는 `approval_first`가 명시되어 있는지 확인한다.
   - 없으면 → 사용자에게 확인받아 컨셉 문서에 기록한 뒤 진행한다.
3. **의도 수집**: 사용자가 원하는 것을 명확히 파악한다. 컨셉 문서와 맞는지 참조한다.
4. **모호성·결정 필요 여부 판단**:
   - 요구사항이 모호하거나 유저 결정이 필요한 부분이 있으면 **task-clarification-interview**를 진행한다.
   - 인터뷰가 끝나기 전에는 스펙·태스크를 구체화하지 않는다.
   - 이미 태스크 초안이 존재하면 `status`를 `waiting_user`로 두고, 없으면 태스크 생성 전까지 보류한다.
   - 이 경우 **"현재 사용자 확인 대기 중"**이라고 안내한다.
5. **스펙 확인**: `docs/specs/`에 해당 기능 스펙이 있는지 확인한다.
   - 있으면 → 기존 스펙 참조.
   - 없으면 → `docs/project-concept.md`를 참조해 범위·우선순위와 맞게 `docs/specs/`에 새 스펙 문서를 생성한다 (planning 규칙 준수: 목표, 요구사항, 수용 조건 포함).
6. **사용자 확인 모드 적용**:
   - 작업 진행 모드가 `approval_first`이면 스펙 초안 또는 태스크 생성 계획을 짧게 요약해 사용자 확인을 먼저 받는다.
   - 답변 전에는 **"현재 사용자 확인 대기 중"**이라고 알리고 태스크 생성으로 넘어가지 않거나, 이미 태스크를 만들었다면 `status`를 `waiting_user`로 둔다.
7. **태스크 생성**: `tasks/items/<id>.md` 파일을 생성한다.
   - `id`: 타입-번호 (예: `feat-002`, `bug-001`). 기존 items를 확인해 번호 중복 방지.
   - `type`: feature / bugfix / design_system / chore 중 적절한 것.
   - `status`: 기본은 `pending`. 단, 사용자 답변·확인이 남아 있으면 `waiting_user`
   - `spec`: 위에서 만든(또는 확인한) 스펙 경로.
   - 본문에 요구사항 요약, 수용 조건을 적는다.
8. **보드 갱신**: `tasks/board.json`의 `items` 배열에 새 태스크 ID를 추가하고, `updatedAt`을 갱신한다.
9. **안내**: "스펙이 생성되었고, pending 태스크가 등록됐습니다." 등 안내.

## 주의

- **컨셉 문서가 없으면** 1단계에서 인터뷰로 전환하고, 스펙·태스크 생성은 컨셉 문서가 채워진 뒤에만 진행한다.
- **요구사항이 모호하면** task-clarification-interview로 전환하고, 답변 전에는 진행하지 않는다.
- 작업 진행 모드가 `approval_first`이면 사용자 확인 전까지 다음 단계로 넘어가지 않는다.
- 태스크 타입에 따라 흐름이 다름: feature는 기획→디자인→개발→테스트, bugfix는 디자인 생략 가능.
- 스펙 작성 시 planning 규칙(`.cursor/rules/planning.mdc`)을 따르며, `docs/project-concept.md`와 범위·용어를 맞춘다.
- 한 번의 의도가 여러 태스크로 분해될 수 있다. 큰 기능이면 하위 태스크로 나눈다.
