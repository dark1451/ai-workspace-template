# 흐름 조율 (Flow Orchestration)

## 트리거

- "보드 상태 봐줘", "다음에 뭘 해야 할지 정리해줘", "병목 있어?", "스프린트 현황", "병렬 에이전트 조율해줘", "coordination 로그 반영해줘"

## 절차

0. **조율 로그(필독)**: `docs/coordination-log.md` 를 연다. **파일 맨 아래**부터 최근 3~5개 블록을 읽어 역할별 마지막 발언·`delta: none` 반복·blocked 여부를 파악한다. 없거나 비어 있으면 한 줄로 넘어간다.
1. **보드 조회**: `tasks/board.json`의 `items`를 읽고, 각 `tasks/items/<id>.md`의 status를 확인한다.
2. **상태별 집계**: pending / in_progress / waiting_user / in_test / done / deployed 각 몇 건인지 집계.
3. **정체·병목 판단**:
   - pending이 많고 in_progress가 0이면 → "개발자/디자이너 픽업 필요".
   - in_test가 쌓이면 → "테스트 에이전트 검증 필요".
   - waiting_user가 쌓이면 → "사용자 확인/의사결정 대기 중인 태스크가 많음"으로 안내.
   - 특정 태스크의 `statusChangedAt`이 3일 이상 전이면 → 정체로 판단. 단, `waiting_user`는 정체가 아니라 사용자 응답 대기로 구분한다.
   - pending을 만들 건이 없고 기획자가 유휴이면 → "아이디에이션 권장".
4. **역할별 권장 액션 도출**:
   - 각 역할(기획자, 디자이너/개발자, 테스트, PM)에 대해 "지금 뭘 해야 하는지" 1줄 요약.
5. **요약 출력**: 상태별 개수 + 권장 액션을 채팅으로 제공한다. **0단계에서 읽은 조율 로그**에서 `delta: none` 반복·blocked·역할별 마지막 발언이 있으면 **한 줄로** 함께 요약한다. 필요 시 `docs/next-actions.md` 갱신은 위 "할 일이 없을 때" 절을 따른다.

## handoff 점검

- in_progress → in_test 전이한 태스크의 메모에 `testCommand`, `runCommand`, `codePaths`, `envNote`가 있는지 확인. 빠져 있으면 보완 제안.

## 피드백 점검

- `docs/feedback/`에 `status: open`인 피드백이 있는지 확인한다.
- open 피드백이 있으면 요약해 사용자에게 알리고 검토를 권장한다.
- blocker 피드백이 있으면 해당 태스크의 진행이 막혀 있음을 강조한다.

## 할 일이 없을 때 → `docs/next-actions.md` 누적 기록

- 점검 결과 모든 역할에 대해 **즉시 권장할 액션이 없거나, 프로젝트 자체가 시작 전**(컨셉 미작성·세팅 미완료)이면, 공통 규칙의 **"할 일 없음 보고 흐름(No-work fallback)"** 4단계를 따른다. PM 만 3번 단계의 누적 기록을 수행한다.
- `docs/next-actions.md` 의 갱신 절차:
  1. 본문 상단의 "마지막 업데이트" 시각·요약을 새 값으로 교체.
  2. 직전 "역할별 권장 액션" 블록을 그대로 "## 변경 이력" 아래로 옮기고 이전 시각을 H3 헤더로.
  3. 새 "역할별 권장 액션" 표에 기획자/디자이너/개발자/테스트/사용자 결정 항목을 1~2줄씩 채운다. 없으면 "—".
  4. 컨셉/세팅이 막혀 있다면 표 위에 사유를 1~3줄 정리하고, **컨셉 인터뷰 시작**을 권장 액션 1순위에 둔다.
- 이 문서는 메인 에이전트가 사용자 "다음 뭐 해?" 같은 질문에 답할 때 **단일 소스로 쓰인다.** 모호한 표현보다 "사용자가 어떤 결정을 내려야 하는지" 를 구체적으로 적는다.

## 인터뷰·todo

- 보드 정리 중 우선순위 재배치·정체 태스크 재할당 등 **사용자 결정이 필요한 항목**이 있으면 `task-clarification-interview` 부터 호출하고, 답 전에는 보드를 임의로 바꾸지 않는다.
- 보드 점검·next-actions 정리가 3단계 이상이면 짧은 todo 리스트(3~7개)로 보여 주며 진행한다 (공통 규칙 "투두 기반 진행").

## 주의

- PM은 태스크를 직접 실행하지 않는다. 보드·흐름 점검, 역할별 다음 액션 정리, 피드백 취합에 집중한다.
