# 태스크 픽업 워크플로우

## 트리거

- "pending 태스크 처리해줘", "다음 태스크 해줘", "개발 진행해줘"

## 절차

1. **pending 조회**: `tasks/board.json`과 `tasks/items/`에서 `status: pending`인 항목을 찾는다. 여러 건이면 `priority`가 높은 것 우선.
2. **컨셉·진행 모드 확인**: `docs/project-concept.md`가 있고, 작업 진행 모드(`proactive` / `approval_first`)와 **신중 모드**(`enabled` / `disabled`), **§5 프로젝트 디자인 컨셉**이 기록되어 있는지 확인한다.
   - 컨셉 문서가 없으면 → 먼저 컨셉 인터뷰를 진행한다.
   - 진행 모드가 없으면 → 사용자에게 확인받아 컨셉 문서에 기록한다.
   - 디자이너 픽업인데 **§5에 사용자에게서 받은 구체 디자인 답변이 없거나, 모든 항목이 "(미정)"이고 한 줄 디자인 방향에도 사용자 답변이 없으면** → 먼저 **task-clarification-interview**를 진행하고, 태스크 `status`를 `waiting_user`로 변경한 뒤 대기한다.
3. **스펙 확인**: 태스크의 `spec` 경로에서 스펙을 읽고 요구사항·수용 조건을 파악한다.
4. **모호성·결정 필요 여부 판단**:
   - 요구사항이 모호하거나 사용자 결정이 필요한 부분이 있으면 **task-clarification-interview**를 진행한다.
   - 답변 전에는 태스크 `status`를 `waiting_user`로 변경하고, 픽업 이후 작업을 진행하지 않으며 **"현재 사용자 확인 대기 중"**이라고 알린다.
5. **사용자 확인 모드 적용**:
   - 작업 진행 모드가 `approval_first`이면 현재 역할에 맞는 시작 게이트를 먼저 통과한다.
   - 디자이너라면 **디자인 착수 전** "이 태스크를 이 범위/방향으로 디자인해도 되는지" 사용자 확인을 받는다.
   - 개발자라면 **구현 시작 전** "이 태스크를 이 범위로 구현해도 되는지" 사용자 확인을 받는다.
   - 확인 전에는 태스크 `status`를 `waiting_user`로 두고 대기하거나, 상황에 따라 픽업 자체를 보류한다.
6. **픽업**: 태스크의 `status`를 `in_progress`로 변경하고 `assignedTo`에 세션 표시를 남긴다. `board.json`도 갱신한다.
7. **역할별 작업 수행**:
   - **디자이너**:
     - **`create-screen-design-spec` 스킬**과 `design/_template-screen-spec.md`를 따라 `design/<task-id>.md`를 작성한다. (색·타이포만 나열 금지)
     - 작업 시작 전 `docs/project-concept.md` **§5 프로젝트 디자인 컨셉**을 읽고, 현재 태스크 디자인안이 그 기준을 어떻게 반영하는지 **"프로젝트 디자인 컨셉 반영 요약"**으로 짧게 정리한다.
     - 프로젝트 디자인 컨셉과 다른 방향이 필요하면 그 이유를 디자인 산출물에 남긴다.
     - 작업 진행 모드가 `approval_first`이면 디자인안 요약과 `design` 경로를 먼저 공유하고 사용자 확인을 받는다.
     - 태스크의 `design` 필드에 경로를 기록한다.
   - **개발자**:
     - `type: feature` UI 태스크인데 `design` 필드에 유효한 `design/*.md`가 없으면 픽업하지 않고 디자인 handoff 대기를 안내한다.
     - `type: feature`이고 태스크의 `design` 필드에 경로가 있으면 **해당 파일(예: design/feat-001.md)을 반드시 읽고**, 그 스펙에 맞게 구현한다.
     - 스펙·디자인에 맞게 코드 작성하고, 해당 범위의 단위/통합 테스트 코드를 작성한다.
     - 태스크 메모에 `codePaths`, `testCommand`, `runCommand`, `envNote`를 기록한다.
8. **신중 모드 셀프리뷰** (§3 신중 모드가 `enabled`일 때): **`self-review` 스킬**에 따라 1회 셀프리뷰를 수행하고, 발견한 수정사항을 반영한 뒤 `selfReviewNote`를 태스크 메모에 기록한다. `disabled`이면 이 단계를 생략한다.
9. **동료 리뷰 요청**: 동일 역할(개발/디자인) 에이전트에게 리뷰를 요청한다.
10. **리뷰 통과 시**:
   - 단일 에이전트 환경에서 별도 동료 리뷰 세션이 없으면, `peer-review` 스킬 규칙에 따라 **사용자 승인**을 리뷰 대체로 사용할 수 있음을 명시하고 진행한다.
   - 사용자 확인이 남아 있으면 `status`를 `waiting_user`로 변경하고 대기한다.
   - **디자이너 작업 완료**라면:
     - 개발 handoff가 필요한 `feature` 태스크는 `design` 링크를 남기고 `status`를 **`pending`**으로 되돌려 개발자가 픽업하게 한다.
     - 순수 디자인 산출물은 프로젝트 규칙에 맞는 다음 단계로 전이한다.
   - **개발자 작업 완료**라면:
     - 작업 진행 모드가 `approval_first`이면 상태 전이 전 사용자 확인을 한 번 더 받는다.
     - 확인되면 `status`를 `in_test`로 변경하고 `board.json`을 갱신한다.
11. **리뷰 수정 요청 시**: 피드백을 반영한 뒤 재리뷰를 요청한다.

## pending이 없을 때

다음 중 우선순위를 정해 작업한다:

1. 버그 수정 (기존 이슈나 코드에서 발견한 것)
2. 테스트 보강 (커버리지 부족한 영역)
3. UI/UX 개선
4. 디자인 시스템 정리·표준화

위 fallback 후보도 마땅한 게 없다면(예: 컨셉·스펙 자체가 없거나, 모든 영역이 충분한 상태) **새로운 작업을 무리하게 만들지 말고** 공통 규칙(`project-defaults` 의 "할 일 없음 보고 흐름") 4단계를 따른다.

1. 응답 머리의 `[역할: ...]` 라벨을 유지한 채 "지금 당장 픽업할 일이 없습니다" 라고 짧게 알린다.
2. "다음에 하면 좋은 일" 후보 3개 이내(예: "기획자가 신규 스펙 1건 발굴", "테스트 회귀 보강", "디자인 시스템 정리")를 정리한다.
3. **PM 역할만** `docs/next-actions.md` 에 누적 기록한다. 개발자·디자이너는 메시지로만 안내.
4. 추가 지시를 기다린다.

## 인터뷰·todo 활용

- 픽업 직후 모호한 점(스펙 해석, 디자인 방향 등)이 있으면 절차 4단계의 `task-clarification-interview` 를 반드시 먼저 수행한다.
- 구현·디자인 산출물이 여러 단계로 늘어나면 응답 시작 때 짧은 todo 리스트(3~7개)를 함께 보여 주고 진행 중에 갱신한다 (공통 규칙 "투두 기반 진행").

## 조율 로그

- 이 워크플로 한 사이클(픽업·완료·대기·No-work 포함)이 끝날 때 **개발자·디자이너**는 `docs/coordination-log.md` 맨 아래에 블록을 추가한다 (`delta`·태스크 ID·한 줄 요약). PM 이 나중에 풀(pull)한다. 형식은 해당 파일 상단 안내·`docs/role-coordination.md` 참조.
