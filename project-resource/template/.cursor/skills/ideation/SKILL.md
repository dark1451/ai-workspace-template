# 아이디에이션

## 트리거

- "아이디에이션 해줘", "기획자 할 일 없으면 아이디에이션 진행해줘"
- 자동화가 "기획자 할 일 없음"으로 호출한 경우

## 절차

1. **현황 파악**: `tasks/board.json`에서 pending 생성 요청이 없는지 확인. `docs/specs/`와 `docs/ideas/`의 기존 항목을 훑어 중복 방지.
2. **아이디어 발굴**: 코드베이스(`apps/`), 기존 스펙(`docs/specs/`), 디자인(`design/`)을 살펴보고 다음 관점에서 후보를 찾는다:
   - 기술 부채·리팩터링 기회
   - UX/UI 개선점
   - 신규 기능 후보
   - 성능·보안 개선
3. **기록**: `docs/ideas/`에 아이디어별 또는 날짜별 파일로 정리한다.
   - 제목, 배경/근거, 예상 효과, (선택) 짧은 스펙 초안
4. **연결**: 나중에 "이 아이디어 스프린트에 넣어줘"라는 요청이 오면 `intent-to-task` 스킬로 스펙·태스크 생성과 연결.

## 주의

- 아이디어는 확정이 아니라 후보이므로, 바로 pending 태스크를 만들지 않는다.
- 기존 아이디어와 겹치는지 확인하고, 겹치면 기존 항목에 보완·병합한다.

## 아이디어 후보도 없을 때

- 코드베이스·스펙·이슈를 살펴봐도 후보가 더 이상 나오지 않으면, 공통 규칙(`project-defaults` 의 "할 일 없음 보고 흐름") 4단계로 마무리한다. 기획자는 `docs/next-actions.md` 누적 기록은 하지 않고 메시지로만 안내한다 (PM 만 기록).

## 인터뷰·todo

- 아이디어 발굴 중 사용자에게 방향(예: 어떤 영역을 우선 볼지)을 물어야 할 때는 `task-clarification-interview` 로 짧게 묻고 진행한다.
- 한 번에 여러 아이디어를 정리할 때는 짧은 todo 리스트(3~7개)로 진행 상황을 보여 준다 (공통 규칙 "투두 기반 진행").

## 조율 로그

- 아이디에이션 세션이 끝나면 `docs/coordination-log.md` 맨 아래에 블록을 추가한다 (`delta`·요약). 형식은 해당 파일 상단·`docs/role-coordination.md` 참조.
