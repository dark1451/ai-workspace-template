# 셀프리뷰 (Self-Review, 신중 모드)

## 트리거

- `docs/project-concept.md` §3 **신중 모드**가 **`enabled`** 일 때
- 에이전트가 **의미 있는 작업 단위**를 1차 완료했을 때 (구현·디자인·스펙·검증·아이디에이션 기록 등)
- 동료 리뷰 요청, handoff, 상태 전이, "작업 완료" 보고 **직전**

## 적용 대상 (모든 역할)

| 역할 | 셀프리뷰 대상 예시 |
|------|-------------------|
| 기획자 | `docs/specs/` 스펙, `docs/ideas/` 아이디어 정리, 태스크 초안 |
| 디자이너 | `design/<task-id>.md`, `design/visual-direction.md` |
| 개발자 | 코드·테스트·handoff 메모 |
| 테스트 | 검증 절차·`testResult` 기록·E2E 시나리오 커버리지 |
| PM | 보드 점검 요약·`next-actions` 초안 (상태 전이를 직접 하지 않는 점검은 생략 가능) |

## 원칙

- **정확히 1회** 셀프리뷰를 수행한다. 건너뛰거나 여러 번 반복하지 않는다.
- 셀프리뷰는 **작성자 관점**에서 산출물·요구사항·컨셉 문서를 다시 대조한다.
- **발견한 수정사항은 즉시 반영**한 뒤, 반영 내용을 기록하고 다음 단계(동료 리뷰·handoff·상태 전이)로 넘어간다.
- 수정할 것이 없으면 "이슈 없음"으로 기록하고 진행한다.
- `careful_mode`가 **`disabled`** 이면 이 스킬을 강제하지 않는다 (기존 peer-review·handoff 흐름만 따름).

## 절차

1. **모드 확인**: `docs/project-concept.md` §3에서 신중 모드가 `enabled`인지 확인한다. `disabled`이면 이 스킬을 종료한다.
2. **기준 재확인**: 해당 태스크의 스펙·수용 조건·`docs/project-concept.md`(특히 §5 디자인 컨셉)·handoff 요구사항을 다시 읽는다.
3. **1회 셀프리뷰 수행**: 아래 역할별 체크리스트로 산출물을 점검한다.
4. **수정 반영**: 지적·개선 항목이 있으면 **바로 수정**한다. 범위를 벗어난 대규모 변경이 필요하면 `docs/feedback/`에 기록하고 사용자에게 알린다.
5. **기록**:
   - 태스크가 있으면 `tasks/items/<id>.md` 메모에 `selfReviewNote`를 남긴다.
   - 태스크 없이 문서만 작성한 경우(아이디에이션 등)는 `docs/coordination-log.md` 블록에 `self-review: pass|fixed` 와 한 줄 요약을 포함한다.
6. **다음 단계**: 셀프리뷰·수정이 끝난 뒤에만 동료 리뷰 요청, handoff, 상태 전이, 완료 보고를 진행한다.

## 역할별 체크리스트

### 기획 (스펙·태스크·아이디어)
- [ ] 목표·수용 조건이 `docs/project-concept.md` 범위와 일치하는가?
- [ ] 구현자·디자이너가 추가 질문 없이 이해할 수 있는가?
- [ ] 용어·우선순위·누락된 AC가 없는가?
- [ ] 태스크 분해·타입·handoff 필드가 적절한가?

### 디자인
- [ ] §5 프로젝트 디자인 컨셉과 방향이 맞는가?
- [ ] 레이아웃·위계·상태·인터랙션이 구현 가능한 수준인가?
- [ ] 색·타이포만 나열한 얇은 handoff가 아닌가?
- [ ] `design` 경로·handoff 메모가 채워졌는가?

### 개발
- [ ] 스펙·디자인(있을 경우)을 준수하는가?
- [ ] 테스트·린트·타입 관점에서 명백한 결함이 없는가?
- [ ] `codePaths`, `testCommand`, `runCommand`, `envNote`가 정확한가?
- [ ] 불필요한 범위 확장·디버그 코드·주석 처리 잔재가 없는가?

### 테스트
- [ ] 수용 조건·handoff의 `testCommand`를 실제로 커버했는가?
- [ ] `testResult`가 판단 근거와 함께 정확히 기록되었는가?
- [ ] 실패 시 되돌릴 상태·사유가 명확한가?

## 기록 형식

태스크 메모 예시:

```yaml
selfReviewNote: "셀프리뷰 1회: AC-2 누락 테스트 추가, handoff testCommand 수정. 이슈 없음 확인 후 in_test 전이."
```

또는 이슈 없을 때:

```yaml
selfReviewNote: "셀프리뷰 1회: 스펙·디자인 대조 완료, 수정 없음."
```

## 주의

- 셀프리뷰는 **동료 리뷰(peer-review)를 대체하지 않는다**. 신중 모드가 켜져 있으면 **셀프리뷰 → (수정) → 동료 리뷰** 순서를 따른다.
- 단일 에이전트 환경에서 peer-review가 셀프 체크리스트로 대체되더라도, 신중 모드의 **1회 셀프리뷰·수정 반영**은 **별도로** 먼저 수행한다.
- 짧은 단일 파일 수정(오타 1건 등)처럼 **의미 있는 작업 단위**가 아니면 생략할 수 있다. 판단이 애매하면 수행하는 쪽을 택한다.
