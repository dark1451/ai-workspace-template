# 프로젝트 컨셉 정의 인터뷰

## 트리거

- 이 템플릿으로 **처음 프로젝트를 진행**하는 유저와 대화할 때
- **`docs/project-concept.md`가 없거나**, 있더라도 **실질적으로 비어 있을 때** (프로젝트명·목표·대상 등이 채워지지 않음)
- 사용자가 "프로젝트 컨셉 정리해줘", "무엇을 만들지 정의해줘" 등으로 요청할 때

## 원칙

- **컨셉 문서가 없으면** 스펙·태스크·디자인·개발을 진행하지 않는다. 먼저 컨셉을 정의한다.
- **인터뷰**는 한 번에 다 묻기보다, 대화 흐름에 맞춰 항목을 채워 나간다. 유저가 답한 내용을 바탕으로 문서를 채우고, 부족한 항목은 자연스럽게 추가 질문한다.
- 한 번의 메시지에서는 가능하면 **1~2개의 질문 묶음**만 던진다. 첫 사용자가 답하기 부담스럽지 않게 유지한다.
- **유저와 나눈 대화**에서 프로젝트 관련 결정·변경이 있으면, `docs/project-concept.md`가 **이미 채워진 상태**에서는 파일을 고치기 전에 반드시 **「문서를 업데이트 할까요?」**라고 묻고, 동의한 뒤 갱신한다 (갱신 이력 기록). 예외는 **공통 규칙 project-defaults**의 "확인 질문 생략"과 같다.
- **작업 진행 모드**는 반드시 받는다. 유저가 `능동 진행 (proactive)`을 원하는지, `확인 우선 (approval_first)`을 원하는지 확인하지 않으면 인터뷰를 완료하지 않는다.
- 사용자가 "작은 파일럿 하나 만들어줘", "일단 시작해줘"처럼 **짧고 비기술적인 요청**만 했더라도, 그 문장에서 프로젝트 유형과 의도를 최대한 추출해 인터뷰를 이어간다.
- 사용자가 작업 진행 모드를 말하지 않은 채 시작만 요청하면, **임시 기본값으로 `approval_first`**를 기록하고 이후 명시 응답이 오면 즉시 갱신한다.
- **프로젝트 디자인 컨셉은 최초 인터뷰에서 반드시 다룬다.** 전문 용어 없이 무드·밝기(라이트/다크)·닮고 싶은 서비스 등으로 질문하고, 답을 **`docs/project-concept.md` §5 프로젝트 디자인 컨셉**에 옮긴다.
- 유저가 "아직 모르겠다"면 그대로 종료하지 말고, **짧은 선택지형 보조 질문**으로 최소 1개의 실제 선호를 받는다. 예: "친근한 편 / 전문적인 편", "밝은 화면 / 어두운 화면 / 둘 다", "카드형 / 리스트형".
- **§5 전체를 "(미정)"으로 둔 채 인터뷰를 완료하지 않는다.** 최소 1개 항목 또는 **한 줄 디자인 방향**에는 사용자에게서 받은 구체 답변이 들어가야 한다.

## 절차

### 1. 컨셉 문서 존재·유효성 확인

- `docs/project-concept.md`가 있는지 확인한다.
- 있으면 내용이 채워져 있는지 본다 (프로젝트 개요, 목표, 대상 사용자, 핵심 기능 등이 구체적으로 있는지).
- **없거나 비어 있으면** 아래 인터뷰를 진행한다.

### 2. 시작 질문

- **"어떤 프로젝트를 진행하시겠습니까?"** (또는 "이번에 만들고 싶은 프로젝트가 어떻게 되나요?")로 시작한다.
- 유저의 첫 답변에서 프로젝트명·한 줄 설명·배경을 최대한 추출하고, `docs/project-concept.md`의 해당 섹션을 채운다.
- 사용자가 이미 프로젝트 유형을 말한 상태라면 질문을 반복하지 말고, **그 답을 기반으로 바로 다음 1~2개 질문**으로 넘어간다.

### 3. 인터뷰 항목 (순서는 대화에 맞게 조절)

아래 항목을 **자연스러운 대화**로 채워 나간다. 한 번에 모두 묻지 않고, 답변에 따라 다음 질문을 이어간다.

| 항목 (project-concept.md 섹션) | 질문 예시 |
|--------------------------------|-----------|
| 프로젝트명·한 줄 설명·배경 | "프로젝트 이름은 어떻게 할까요?", "왜 이걸 만들고 싶으신가요?" |
| 목표·성공 기준 | "이 프로젝트로 꼭 이루고 싶은 목표가 있나요?", "어떤 상태가 되면 성공이라고 보시나요?" |
| 작업 진행 모드·사용자 확인 정책 | "에이전트가 컨펌 없이 능동적으로 진행해도 될까요, 아니면 단계마다 확인을 받고 진행할까요?", "어느 수준의 변경은 꼭 확인받길 원하시나요?" |
| 대상 사용자·맥락·니즈 | "주로 누가 사용할 예정인가요?", "어떤 상황에서 쓰게 될까요?" |
| **프로젝트 디자인 컨셉** (§5) | "전체적으로 어떤 분위기·느낌이면 좋을까요?", "밝은 화면과 어두운 화면 중 선호가 있나요?", "닮고 싶은 앱·사이트나 피하고 싶은 UI 스타일이 있나요?" (1~2개씩 나눠 묻기) |
| 핵심 기능·MVP 범위 | "MVP에 꼭 넣고 싶은 기능은?", "당장은 하지 않을 기능이 있다면?" |
| 비기능 요구사항 | "성능·보안·플랫폼(웹/모바일)에서 요구사항이 있나요?" |
| 제약·선호 | "쓰고 싶은/쓰지 않을 기술이 있나요?" (디자인 톤·레퍼런스는 §5에서 다룬 뒤, 여기서는 기술·일정 위주로 보완) |
| 용어·정의 | 도메인 용어가 있으면 "OO는 정확히 무엇을 의미하나요?"로 정의를 받아 문서에 적는다. |

- 답변이 모호하면 한두 번만 구체적으로 여쭤본다. 압박하지 않는다.
- **답변을 받을 때마다** `docs/project-concept.md`의 해당 섹션을 갱신한다. 마지막 갱신 날짜와 "컨셉 문서 변경 이력"을 남긴다.

### 4. 문서 작성·갱신

- 인터뷰가 끝나거나, 한 단락씩 채울 때마다 **`docs/project-concept.md`**를 저장한다.
- 템플릿 구조는 `docs/project-concept.md` 상단 안내와 섹션을 따른다. 채울 수 없는 항목은 "(미정)" 또는 "(추후 보완)"으로 두어도 된다.
- **자세할수록 좋다**: 스펙·태스크 구체화에 이 문서가 쓰이므로, 구현자가 읽고 "무엇을 만들어야 하는지" 알 수 있을 정도로 쓴다.
- 특히 **작업 진행 방식·사용자 확인 정책** 섹션은 반드시 채운다. 이 값이 없으면 이후 에이전트가 행동 기준을 정할 수 없다.
- **목표·성공 기준**이 비어 있으면 컨셉 확정 질문으로 넘어가지 않는다. 최소한 "이 프로젝트의 목적"과 "성공이라고 볼 기준"을 한 번은 확인한다.
- **프로젝트 디자인 컨셉(§5)**은 표 전체가 채워지지 않아도 되지만, **최소 1개 이상의 실제 사용자 답변**이 들어가야 한다.
- 유저가 처음 답을 어려워하면, 추상 질문 대신 **선택지형 질문 1~2개**로 좁혀 다시 묻는다.
- 답변이 아직 없으면 `docs/project-concept.md`를 임시 저장하더라도 컨셉 인터뷰를 완료하지 말고, 필요한 질문을 짧게 제시한 뒤 **"현재 사용자 확인 대기 중"**이라고 알린다.

### 5. 대화 중 변경사항 반영

- **이후 대화**에서 "그 기능은 빼자", "우선순위를 바꾸자", "대상 사용자를 OO로 한정하자" 등 **프로젝트 관련 결정·변경**이 나오면:
  - 먼저 **「문서를 업데이트 할까요?」**라고 묻고, 반영할 항목을 짧게 요약한 뒤 사용자가 동의하면
  - 해당 내용을 **`docs/project-concept.md`**에 반영하고,
  - **"컨셉 문서 변경 이력"**에 날짜와 변경 요약을 추가한다.
  - 같은 턴에 갱신을 명시했거나, 컨셉 문서가 없거나 비어 있어 **최초 인터뷰만** 진행 중인 경우 등 **예외**는 project-defaults 규칙을 따른다.

## 완료 조건

- `docs/project-concept.md`에 **프로젝트 개요, 목표, 작업 진행 모드, 대상 사용자, 프로젝트 디자인 컨셉(§5에 최소 1개 이상의 실제 사용자 답변 포함), MVP 핵심 기능**이 채워져 있어, 에이전트가 "무엇을 만들어야 하는지", "어떤 느낌으로 보여야 하는지", "어떻게 진행해야 하는지" 판단할 수 있는 수준이 되면 인터뷰를 마무리해도 된다.
- 유저에게 "컨셉 문서를 이렇게 정리해 두었습니다. 더 보완할 부분이 있으면 말씀해 주세요. 확인해 주시면 다음으로 `docs/specs/`에 MVP 스펙 초안을 정리하겠습니다."처럼 **다음 산출물**을 함께 안내한다.

## 주의

- 컨셉 문서가 채워지기 전에는 **intent-to-task**(스펙·태스크 생성)나 **task-pickup**(구현)을 진행하지 않는다. 먼저 이 스킬로 컨셉을 채운다.
- `docs/project-concept-README.md`에 컨셉 문서의 목적·갱신 원칙이 정리되어 있으므로, 필요 시 참고한다.
