# 에이전트 피드백 (Feedback / Complaints)

에이전트가 작업 중 발견한 문제점·우려·개선 제안을 기록하는 곳이다.

## 목적

- 에이전트가 "이건 문제가 있다"는 것을 표현할 수 있는 공식 채널
- 축적된 피드백을 사용자가 검토하고, 규칙·스펙·프로세스를 개선하는 근거로 사용
- 문제가 발견된 시점에 즉시 기록하여 컨텍스트가 유실되지 않도록 함

## 피드백 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| `spec-unclear` | 스펙·요구사항이 불명확 | "수용 조건이 모호해 구현 범위를 판단할 수 없음" |
| `rule-conflict` | 규칙 간 충돌·모순 | "development.mdc와 testing.mdc globs가 겹침" |
| `process-bottleneck` | 프로세스 병목·비효율 | "동료 리뷰가 교착됨" |
| `tech-concern` | 기술적 우려·부채 | "supabase 클라이언트에 에러 핸들링 없음" |
| `improvement` | 개선 제안 | "태스크에 예상 소요 시간 필드 추가하면 좋겠음" |
| `blocker` | 작업 진행 불가 | "env 변수 미설정으로 테스트 실행 불가" |

## 파일 형식

`docs/feedback/YYYY-MM-DD-<짧은설명>.md` 또는 주제별 파일:

```markdown
---
type: spec-unclear
severity: medium
taskId: feat-001
role: developer
date: "2026-03-13"
status: open
---

# [피드백 제목]

## 상황

(어떤 작업 중에, 무엇을 하려다 발견했는지)

## 문제

(구체적으로 무엇이 문제인지)

## 제안

(가능하면 해결 방안 또는 대안)

## 사용자 결정

(사용자가 검토 후 기록: 수용 / 기각 / 보류 + 사유)
```

## 상태

| 상태 | 설명 |
|------|------|
| `open` | 에이전트가 제출, 사용자 검토 대기 |
| `acknowledged` | 사용자가 확인, 조치 예정 |
| `resolved` | 조치 완료 (규칙 수정, 스펙 보완 등) |
| `wont-fix` | 의도된 동작 또는 현재로서는 수용 |

## 규칙

- 에이전트는 작업 중 문제를 발견하면 **즉시** 피드백을 기록해야 한다 (작업을 멈출 필요는 없음).
- 심각도가 `blocker`이면 작업을 중단하고 사용자에게 알린다.
- 사용자는 주기적으로 `docs/feedback/`을 검토하고 `status`를 갱신한다.
- 해결된 피드백은 해당 규칙·스펙·프로세스 문서에 반영하고, 피드백 파일의 status를 `resolved`로 변경한다.
