## 3. 작업 진행 방식·사용자 확인 정책

| 항목 | 내용 |
|------|------|
| **작업 진행 모드** | `능동 진행 (proactive)` / `확인 우선 (approval_first)` 중 하나를 반드시 선택 |
| **세부 원칙** | (예: 큰 단계만 확인, 모든 UI 변경 확인, 데이터 구조 변경은 항상 확인 등) |

- **능동 진행 (proactive)**:
  - 컨셉과 스펙이 명확하면 에이전트가 다음 작업을 능동적으로 진행할 수 있다.
  - 단, **모호한 점**이나 **유저 결정이 필요한 사항**이 있으면 반드시 인터뷰 후 진행한다.
- **확인 우선 (approval_first)**:
  - 에이전트는 **의미 있는 다음 단계로 넘어가기 전** 사용자 확인을 받아야 한다.
  - 예: 컨셉 확정, 스펙 생성, 태스크 등록, 구현 시작, 상태 전이 전.
- **대기 절차**:
  1. 확인이 필요한 쟁점이나 선택지를 짧게 정리한다.
  2. 유저에게 질문하거나 선택지를 제시한다.
  3. **"현재 사용자 확인 대기 중"**이라고 명시하고, 답변 전까지 다음 작업을 진행하지 않는다.
- **모호성 처리 원칙**:
  - 태스크 진행 중 모호한 점, 상충하는 해석, 유저 취향/비즈니스 판단이 필요한 부분이 있으면 **태스크 명확화 인터뷰**를 먼저 진행한다.
  - 이 인터뷰가 끝나기 전에는 스펙·태스크를 구체화하거나 구현을 진행하지 않는다.

## 5. 프로젝트 디자인 컨셉

> 프로젝트 전반의 시각적 기준이다. 최초 컨셉 인터뷰에서 **비개발자도 이해할 수 있는 말**로 받아 적는다. (예: "밝고 친근한 느낌", "은행 앱처럼 신뢰감")  
> 디자이너·개발자는 태스크용 `design/` 산출물을 만들 때 이 섹션을 **우선 참고**한다. 태스크별 디자인안이 이 기준과 다르면, 왜 다른지 handoff에 짧게 남긴다.
> **이 섹션은 전부 "(미정)"으로만 두고 넘어가지 않는다.** 최소 1개 이상의 항목 또는 **한 줄 디자인 방향**에는 사용자에게서 받은 실제 답변이 들어가야 한다.

| 항목 | 내용 |
|------|------|
| **전체 무드·톤** | (예: 친근 / 전문 / 미니멀 / 플레이풀 / 럭셔리 등, 한두 문장) |
| **라이트·다크·자동** | (선호: 라이트만 / 다크 우선 / 둘 다 / 시스템 연동 등) |
| **밀도·레이아웃 성향** | (여백 많은 편 / 정보 밀도 높음 / 카드형 / 리스트형 등) |
| **색·브랜드** | (브랜드 컬러 있음·없음, 피하고 싶은 색, 참고 팔레트가 있으면 요약) |
| **타이포·가독성** | (한글 위주 / 영문 비중 큼, 큰 글씨 선호, 장식적 폰트 OK 여부 등) |
| **참고·레퍼런스** | (닮고 싶은 서비스·앱, 싫은 UI 예시, 피그마/이미지 링크는 §10에도 가능) |
| **접근성·모션** | (애니메이션 많이 / 최소, 고대비 필요, 키보드 위주 등 — 없으면 "(미정)") |

- **한 줄 디자인 방향**: (위를 요약한 한 문장 — 디자이너가 태스크 픽업 시 가장 먼저 보는 기준)
