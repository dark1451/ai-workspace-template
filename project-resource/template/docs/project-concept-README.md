# 프로젝트 컨셉 문서

## 목적

이 템플릿으로 **처음 프로젝트를 진행하는 유저**는, 에이전트가 무엇을 만들어야 할지 알 수 있도록 **프로젝트 컨셉 문서**를 먼저 정의해야 합니다.

- **에이전트**: "어떤 프로젝트를 진행하시겠습니까?"라고 묻고, 이에 대한 대화(인터뷰)를 통해 컨셉 문서를 채웁니다.
- **컨셉 문서**: `docs/project-concept.md` 한 편이 해당 프로젝트의 **단일 기준(single source of truth)** 이 됩니다.
- **산출물 기준**: 스펙·디자인·태스크·구현은 모두 이 컨셉 문서를 기반으로 합니다. **컨셉 문서가 없거나 비어 있으면 본격적인 작업(스펙 작성, 태스크 생성, 디자인, 개발)을 진행하지 않습니다.**
- **작업 진행 모드 기준**: 컨셉 문서에는 **`능동 진행 (proactive)`** 또는 **`확인 우선 (approval_first)`** 중 하나를 반드시 기록합니다. 이후 모든 에이전트는 이 값을 기준으로 사용자 확인 여부를 결정합니다.
- **짧은 요청도 가능**: 사용자가 "작게 시작해줘", "파일럿 하나 만들어줘"처럼 비기술적인 한두 문장만 말해도, 에이전트는 현재 문서와 보드를 읽고 필요한 컨셉 질문으로 이어갑니다.
- **모드 미정 시 기본값**: 사용자가 작업 진행 모드를 아직 말하지 않았다면, 에이전트는 안전하게 **`approval_first`**를 임시 기본값으로 두고 진행합니다.

## 이 문서로 하는 일

1. **무엇을 만들지 구체화**: 목표, 사용자, MVP 기능, 제약을 명시해 "무엇을 만들어야 하는지"를 한곳에 정의합니다.
2. **태스크 구체화**: 기획·스펙·태스크 분해 시 이 문서를 참조해 일관된 범위와 우선순위로 작업합니다.
3. **역할 간 공유**: 기획자·디자이너·개발자·테스트 에이전트가 같은 컨셉을 바라보고 작업할 수 있게 합니다.
4. **진행 방식 통일**: 유저가 "능동 진행"을 원하는지, "확인 우선"을 원하는지 기준을 명시해 에이전트의 행동 방식을 통일합니다.

## 작업 진행 모드

- **능동 진행 (`proactive`)**:
  - 컨셉과 스펙이 충분히 명확하면 에이전트가 다음 작업을 자율적으로 진행합니다.
  - 단, **모호한 점**이나 **유저 결정이 필요한 부분**이 있으면 반드시 질문하고 답변을 받은 뒤 진행합니다.
- **확인 우선 (`approval_first`)**:
  - 에이전트는 의미 있는 다음 단계로 넘어가기 전에 사용자 확인을 받습니다.
  - 예: 스펙 생성, 태스크 등록, 구현 시작, 상태 전이 전.
- **대기 절차**:
  1. 확인이 필요한 항목을 짧게 정리합니다.
  2. 질문 또는 선택지를 제시합니다.
  3. **"현재 사용자 확인 대기 중"**이라고 명시하고 답변 전까지 진행하지 않습니다.

## 모호성·결정 필요 시 인터뷰

- 태스크를 진행하는 도중에도 모호한 요구사항, 여러 해석이 가능한 부분, 유저 취향/비즈니스 판단이 필요한 선택이 나오면 **짧은 명확화 인터뷰**를 먼저 진행합니다.
- 이 인터뷰가 끝나기 전에는 스펙·태스크를 더 구체화하거나 구현을 이어가지 않습니다.
- 인터뷰 결과는 `docs/project-concept.md` 또는 관련 스펙/태스크 문서에 반영합니다.
- 사용자가 `waiting_user` 상태에서 "응", "좋아", "그 방향으로", "후순위로"처럼 짧게 답하더라도, 직전 질문이 명확했다면 그 응답을 유효한 승인/선택으로 보고 다음 단계로 재개할 수 있습니다.

## 자세히 쓸수록 좋은 이유

- 스펙·태스크가 컨셉과 어긋나지 않도록 할 수 있습니다.
- "이건 포함/제외" 같은 경계가 명확해집니다.
- 용어·정의를 통일해 오해를 줄일 수 있습니다.

## 지속적인 갱신

- **유저와 나눈 대화**에서 프로젝트 관련 결정·변경(기능 추가/삭제, 우선순위 변경, 제약 사항 변경 등)이 있으면 **컨셉 문서를 갱신**합니다.
- 특히 **작업 진행 모드**, 승인 범위, 유저가 직접 결정하길 원하는 항목이 바뀌면 반드시 컨셉 문서에 반영합니다.
- 갱신 시 "컨셉 문서 변경 이력" 섹션에 날짜와 변경 요약을 남기면, 이후 에이전트·유저가 흐름을 추적하기 쉽습니다.
- 누가 갱신하든: 대화를 진행한 에이전트가 `docs/project-concept.md`를 수정하는 것을 원칙으로 합니다.

## 관련 규칙·스킬

- **공통 규칙** (`.cursor/rules/project-defaults.mdc`): 컨셉 문서 없으면 본격 작업 전에 컨셉 인터뷰를 진행하도록 되어 있습니다.
- **스킬** (`.cursor/skills/project-concept-interview/`): "어떤 프로젝트를 진행하시겠습니까?" → 인터뷰 → 문서 작성·갱신 절차를 따릅니다.
- **스킬** (`.cursor/skills/task-clarification-interview/`): 태스크 진행 중 모호한 점이나 사용자 결정이 필요한 사항이 생기면 명확화 인터뷰를 진행하고, 답변 전까지 대기하도록 합니다.
