# 롤 에이전트 병렬 실행과 조율 (Role coordination)

이 템플릿은 **여러 Cursor / `cursor-agent` 세션**에서 기획·디자인·개발·테스트·PM 역할을 **동시에 또는 연속으로** 돌릴 수 있도록 설계되어 있다. 다만 세션끼리는 **채팅으로 서로 말을 걸 수 없다**. 따라서 “PM에게 결과 요약을 자동으로 취합한다”는 **실시간 브로드캐스트**는 없다.

대신 아래 **파일 기반 비동기 조율**으로 “누가 무엇을 했는지 / 막혔는지”를 공유한다.

## 현재 세팅 vs 향후 실시간 소통 (Slack·Teams 등)

### 지금 이 템플릿에 기본으로 깔려 있는 것

- **에이전트 세션끼리 채팅·실시간 푸시 없음.** Cursor IDE 채팅 여러 탭이든, `cursor-agent` CLI 여러 개든, 서로 **자동으로 메시지를 주고받지 않는다.**
- 조율·취합·“뭐 했는지”는 전부 **같은 워크스페이스 안의 파일**로만 한다: `tasks/board.json` + `tasks/items/*.md`, `docs/coordination-log.md`, `docs/next-actions.md`(PM), `docs/feedback/`.
- PM이 “알게” 되는 경로도 **파일을 읽는 풀(pull)** 이다. 사용자가 PM 세션을 돌리거나 “보드 점검해줘”라고 할 때 `flow-orchestration` 절차가 위 파일을 연다.

### 나중에 Slack 등으로 실시간 검토·소통을 붙일 때

- **가능하다.** 다만 이 저장소의 규칙·스킬이 **Slack API에 자동 연결되어 있지는 않다.** 연동(웹훅, 봇, 사람이 채널 요약을 복사해 `coordination-log`에 붙이기 등)은 **팀이 추가**하는 영역이다.
- 운영을 안정적으로 가져가려면 채널에서는 **대화·알림·결정 문장**을 하고, **최종 상태는 여전히 저장소 파일**에 반영하는 패턴을 권장한다. (채널에만 결정이 남고 보드·로그가 비면, 에이전트와 PM이 같은 그림을 못 본다.)
- 예: Slack에서 “이 태스크 승인” → 사람 또는 봇이 `tasks/items/…`·`coordination-log`·필요 시 `next-actions`를 갱신 → 다음에 돌리는 Cursor/CLI 세션이 그걸 읽고 진행.

요약하면, **지금은 파일·Git 중심 비동기**, **나중에 Slack 등은 사람 중심 실시간 레이어로 얹을 수 있고**, 단일 진실은 계속 **리포 안 문서·보드**에 두는 것이 이 템플릿의 전제다.

## 1. 단일 진실 공급원 (Single sources of truth)

| 용도 | 위치 | 누가 쓰나 |
|------|------|-----------|
| 작업 큐·상태 | `tasks/board.json` + `tasks/items/<id>.md` | 모든 롤. 상태·핸드오프 메모는 **여기가 최우선**이다. |
| 역할 세션 요약·막힘·델타 없음 | `docs/coordination-log.md` | **모든 롤**이 세션 단위로 **맨 아래에 블록 추가**. PM·메인 에이전트가 **먼저 읽는다**. |
| PM 종합·“다음에 뭐 할지” | `docs/next-actions.md` | **PM 전용**으로 갱신. 전체가 유휴할 때·사용자가 방향을 물 때의 요약 소스. |
| 블로커·스펙 이슈 | `docs/feedback/` | 모든 롤. `blocker`면 작업 중단 + 사용자 알림. |

**PM은 다른 세션을 “구독”하지 않는다.** PM 세션을 돌릴 때 위 파일들을 **풀**해서 최신 상태를 재구성한다.

## 2. “PM에게 취합”이 의미하는 것

- **푸시(push)**: 롤 에이전트가 PM 프로세스를 깨우는 것 — 이 저장소 규칙만으로는 **불가능**하다.
- **풀(pull)**: 사용자가 `pnpm test:scenario:pm` 등으로 PM을 실행하거나, 메인 채팅에서 “보드 점검해줘”라고 할 때 PM이 **로그·보드를 읽고** 요약하는 것 — **이게 공식 취합 경로**다.

따라서 장기·병렬 운영에서는 **PM 주기**(예: 하루 한 번, 스프린트 시작/종료, 병렬 세션 묶음이 끝날 때마다)를 팀이 정해 두는 것이 좋다.

## 3. 롤 에이전트가 반드시 남기는 것 (`docs/coordination-log.md`)

**세션이 끝나거나**, 사용자와 한 덩어리의 작업이 끝날 때(또는 **보드/코드/문서에 변화가 없는 채** 같은 롤로만 대화가 이어질 때) **맨 아래에 한 블록**을 추가한다.

- **진행이 있었으면**: 픽업한 태스크 ID, 상태 변경, 변경 파일 요약, 다음에게 넘길 것.
- **진행이 없었으면** (`delta: none`): 왜 없었는지(예: `pending` 없음, 컨셉 비어 있음, `waiting_user`, 환경 미설정), PM/사용자가 할 일 한 줄.

이렇게 해야 “개발자 롤만 계속 켜져 있는데 아무 일도 안 일어난다”가 **로그와 보드에 남고**, PM이 다음 실행에서 **인지**할 수 있다.

## 4. PM이 병렬 상황에서 하는 일

`flow-orchestration` 스킬 순서에 맞춰, 대략 다음을 한다.

1. `docs/coordination-log.md` **최근 블록**부터 읽기 (역할별 마지막 발언 확인).
2. `tasks/board.json` + `tasks/items/` 집계 (`in_progress` 장기 정체는 coordination 내용과 교차 확인).
3. `docs/next-actions.md`, `docs/feedback/` (open / blocker).

출력은 “상태 숫자 + 역할별 한 줄 + 막힌 것 + 사용자에게 물을 것”으로 정리한다.

**선택(양방향 보강)**: 주요 점검 직후 PM이 `docs/coordination-log.md` 맨 아래에 **짧은 한 블록**을 남겨도 된다(`[역할: PM]`). 다른 병렬 세션이 PM 실행을 놓쳐도 “마지막으로 PM이 본 그림”을 공유할 수 있다. 전체 요약·다음 행동은 여전히 `docs/next-actions.md` 가 우선이다.

## 5. 양방향 소통이란?

- **롤 → PM/타 롤**: 태스크 메모, `coordination-log`, 필요 시 `docs/feedback/`. (코드·스펙·디자인 파일 포함.)
- **PM → 롤**: `docs/next-actions.md`의 역할별 권장 액션, 보드에서의 상태·우선순위 정리, 사용자에게 요청할 결정 문장. (선택) PM이 `coordination-log`에 `[역할: PM]` 한 블록을 남기면 롤 세션이 PM 실행 시점을 놓쳐도 맥락을 잡을 수 있다.
- **사용자 → 전체**: 메인 세션에서 “PM 돌려줘”, “개발자 픽업해줘”처럼 트리거를 주면, 각 세션이 위 파일을 읽고 **같은 그림**으로 맞춘다.

## 6. 관련 규칙·스킬

- 공통: `.cursor/rules/project-defaults.mdc` — **병렬 롤 에이전트·비동기 조율**, **coordination-log 의무**, **진행 델타 없음** 처리.
- PM: `.cursor/rules/pm.mdc`, `flow-orchestration` 스킬 — 조율 시 coordination-log 필독.
- 실제 기록 파일: `docs/coordination-log.md` (이 문서와 짝).
